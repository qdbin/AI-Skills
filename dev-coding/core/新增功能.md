## 需求理解检查清单

## 设计自检要点
设计方案时，请按以下要点进行自检：
### 架构设计
- [ ] 是否遵循开闭原则？新增功能是否对原有代码开放扩展、关闭修改
- [ ] 模块划分是否清晰？职责是否单一？
- [ ] 是否与现有模块解耦？依赖是否明确？

### 接口设计
- [ ] 接口命名是否清晰、见名知意？
- [ ] 参数设计是否精准解耦？是否避免与业务强绑定？
- [ ] 返回值设计是否完整？异常情况是否考虑？

### 扩展性
- [ ] 是否预留了扩展点？便于后续功能扩展
- [ ] 是否考虑了未来可能的变化？

### 改动评估
- [ ] 预估改动范围：涉及哪些文件？影响哪些模块？
- [ ] 是否需要同步修改关联功能？
- [ ] 是否有现成的工具/方法可复用？


## 实施前方案评估
- [ ] 改动量是否最小化？
- [ ] 是否为最优方案？是否有更优方案？
- [ ] 是否充分考虑了安全性？（如权限校验、敏感数据处理）


## 实施后检查
- [ ] 完成后是否进行了自测？
- [ ] 是否运行了相关单元测试？
- [ ] 是否清理了调试代码？
- [ ] 是否更新了相关注释？


## 现有要求
- 其他已有的无关功能保持不变
- 充分理解当前系统及其他功能模块，自行合理处理相关联的功能实现（即对于部分受影响的小功能）
- 务必充分理解当前背景和目标功能实现！！！
- 只补充缺失功能函数，不可修改和优化无关的功能，务必遵循开闭原则！！！！！！
- 只添加新功能，若无明确要求，不要对主要逻辑、已有功能、架构及其他源码进行任何改动和优化！！！
- 实施最优方案：有明确的方案后，先不要着急实施，再想想是否可优化或其他方案，能使该其改动更少的最优方案
