编码开发的规范和要求
---

可依据`$技术方案`编码实现，保证不偏离需求的前提，动态优化，可以摒弃其中不完善、不合理、错误分析、设计偏差的点

## 要求须知
### 原则规范
- 务必严格依据该 `$脚本设计要求&规范&原则：{{  }}` 的要求和规范，以设计实现目标脚本，务必遵守其要求&规范，确保其复用、解耦与维护
- 面向对象封装，不同的执行流程应采用不同的顶层封装函数，编码逻辑清晰
- 模块独立解耦、开闭原则、职责单一，保持高通用&高复用，尽可能确保每个方法功能独立解耦

### 规范编码
- `类名`&`函数名`&`局部变量名`等命名尽可能通用、专业、规范（适当泛化语义命名，便于复用到其他脚本中）
- 对高复用易迁移的工具函数方法
  - 精准解耦的形参设计，非必要则不要与业务挂钩，提高通用、复用与迁移性，避免迁移维护成本过大
  - 其次适当宽泛的专业命名，提高通用性、易读性
- 务必将`可读性`和`可维护性`作为编码首要目标之一，确保代码高质量（独立解耦，不冗余）
- 合理控制函数功能范围，职责单一，由下向上层层封装（务必合理控制功能单元范围，不可太琐碎）
- 将主要实现步骤阶段，抽离出来单独封装，每次扩展功能，只需再封装一个使逻辑清晰
- 为主要步骤及阶段添加注释的同时，也有有必要的序号标注（如：一、 二、 三、 …、 1、2、3、… a、b、c、…）

### 细节处理
- 关键步骤需`print打印输出`（如执行阶段），特殊数据结构需用`pprint打印输出`
- 异常处理：抛出的异常要有message（如若脚本执行因arg依赖问题或不服要求）


## 验证机制
- 统一撰写单元测试、自行调试执行并验证每个功能按预期实现，务必确保功能符合预期



## 编码自检清单（完成后必查）
完成编码后，请按以下要点进行自检：
## 基本要求
- 遵循开闭原则、复用性，模块独立性，专业性，规范性
- 删除冗余且无太大意义的代码和方法
- 对重要阶段与步骤要有`空行`+`行首`注释！！！


### 原则检查
- [ ] 是否遵循开闭原则：对扩展开放，对修改关闭
- [ ] 是否遵循职责单一原则：每个函数/类只做一件事
- [ ] 模块是否独立解耦：功能之间无强依赖

### 命名检查
- [ ] 类名、函数名、变量名是否专业规范、见名知意
- [ ] 是否进行了适当的语义泛化，便于复用
- [ ] 工具类方法的形参是否精准解耦，非必要不与业务挂钩

### 可读性检查
- [ ] 注释率是否达到30%
- [ ] 关键步骤是否有空行+行首注释
- [ ] 难点代码是否有同行末尾注释

### 功能检查
- [ ] 异常处理是否包含明确的错误message
- [ ] 单元测试是否覆盖核心功能


- [ ] 确保未提交敏感信息（如密钥、密码、token等）
