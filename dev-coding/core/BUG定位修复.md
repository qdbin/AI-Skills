【Case3】 BUG 定位与修复，请遵循先验证、后修复的开发规范
---

当发现代码存在问题时，请你先不要直接修改代码，请优先通过增加调试日志、编写独立测试代码/测试脚本的方式复现与定位问题：
- Python 中可在 if __name__ == '__main__' 块内编写调用验证逻辑
- Java/SpringBoot：在 src/test/java 目录下编写对应的单元测试类进行验证
先通过测试代码复现、定位问题，确认问题根因后，再给出最小、安全的代码修改方案，确保调试代码与业务代码解耦，不污染主逻辑。
- 当然也可以调试日志 / 日志埋点（print、log）,最后再统一删除即可

## 要求
- 不允许拍脑袋修改
- 不允许引入新的回归问题
- 不允许修改无关功能模块
- 务必充分理解当前背景和相关功能实现，自行合理处理相关联的功能实现（即对于部分受影响的小功能）
- 务必精准定位有缺陷功能并精准最优修复，不可修改和优化无关的功能，务必遵循开闭原则！！！！！！
- 实施最优方案：有明确的方案后，先不要着急实施，再想想是否可优化或其他方案，能使该其改动更少的最优方案

## 常见Bug类型分类总结
### 1. 空指针/空值类
- NullPointerException（空指针异常）
- 空列表/空字典访问异常
- 未初始化变量使用

### 2. 类型相关类
- 类型转换异常（ClassCastException）
- 类型不匹配（TypeError）
- 隐式类型转换导致的精度丢失

### 3. 并发相关类
- 线程安全导致的死锁
- 竞态条件（Race Condition）
- 共享资源未加锁

### 4. 逻辑错误类
- 边界条件处理错误
- 循环边界错误（off-by-one）
- 判断条件写反

### 5. 资源泄漏类
- 数据库连接未关闭
- 文件流未关闭
- 内存泄漏

### 6. 边界条件类
- 数组下标越界
- 除零错误
- 超大/超小数值处理


## 定位问题常用手段
### 1. 日志分析
- 查看错误日志的堆栈信息，定位异常抛出位置
- 分析关键变量的值是否符合预期
- 对比正常流程和异常流程的日志差异

### 2. 断点调试
- 在可疑位置设置断点
- 逐步执行观察变量变化
- 重点关注条件分支和循环边界

### 3. 最小复现
- 构造最简单的复现用例
- 逐步简化输入，直到找到触发条件
- 记录每次尝试的结果

### 4. 二分定位法
- 通过注释或删除部分代码，一半一半排查
- 快速缩小问题范围到某个代码块

### 5. 对比分析法
- 与正常功能的相似代码进行对比
- 查找两者之间的差异点


## 修复后必做清单
- [ ] 使用相同测试用例验证修复成功
- [ ] 运行相关模块的单元测试，确保无回归
- [ ] 清理调试日志和临时测试代码
- [ ] 检查是否影响关联功能（如有则同步修复）
- [ ] 更新相关注释（如问题原因已变化）
